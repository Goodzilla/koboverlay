## Authentication Flow (Twitch OAuth + JWT)

KobOverlay uses Twitch OAuth 2.0 for authentication. There are no passwords or email addresses stored.

```
User clicks "Sign in with Twitch"
        |
        v
GET /api/auth/twitch
        | Redirects to https://id.twitch.tv/oauth2/authorize
        v
Twitch Authorization Page
        | User authorizes KobOverlay
        v
GET /api/auth/twitch/callback?code=...
        | Server exchanges code for access token
        | Fetches Twitch user profile (id, login, display_name, profile_image_url)
        | Upserts User record in database (create on first login, update on return)
        | Issues signed JWT (24h expiry)
        v
Redirect to CLIENT_URL/studio?jwt=<token>
        | Client stores JWT in localStorage
        | Client decodes payload to read overlayToken, username, displayName
        v
Studio loaded â€” user is authenticated
```

### JWT Payload Structure
```typescript
{
  userId: string;          // internal DB user ID
  overlayToken: string;    // unique token for OBS browser source URL
  username: string;        // Twitch login name (lowercase)
  displayName: string;     // Twitch display name
  profileImage: string;    // Twitch profile image URL
}
```

### Session Rehydration
On page load, the client reads the JWT from `localStorage` and calls `GET /api/auth/me` with `Authorization: Bearer <jwt>`. If the token is valid and not expired, the user session is restored. If expired or missing, the user is shown the login modal.

## Socket.io Real-Time Protocol

Client overlay instances join isolated room channels based on their unique `overlayToken`:

### 1. Room Joining
- **Event**: `join-overlay`
- **Payload**: `{ token: string }`
- **Behavior**: Server places the socket client into room `overlay:<token>`.

### 2. Alert Trigger Broadcast
- **Event**: `trigger-alert`
- **Payload**:
```typescript
interface AlertEvent {
  id: string;
  type: 'sub' | 'resub' | 'subgift' | 'bits' | 'raid';
  username: string;
  tier: 'Prime' | '1000' | '2000' | '3000';
  months?: number;
  amount?: number;
  message?: string;
  durationMs: number;
  primaryColor: string;
}
```

### 3. Sub Goal Progress Update
- **Event**: `update-sub-goal`
- **Payload**:
```typescript
interface SubGoalUpdate {
  title: string;
  currentSubs: number;
  targetSubs: number;
}
```

### 4. Studio State Persistence & Synchronization
- **Event**: `save-studio-state`
- **Payload**: `{ token: string, widgets: WidgetInstance[] }`
- **Behavior**: Broadcasts `studio-state-updated` to room `overlay:<token>` and upserts the full serialized `widgets` array into `OverlayConfig.layoutConfig` in the database via Prisma.

---

## Database Schema & Cloud Persistence (Prisma)

- **`User`**: Streamer account identified by `twitchId`. Holds `overlayToken` (UUID used in OBS browser source URL), Twitch profile fields, and optional email.
- **`SharedModToken`**: 1-year max shared links generated by the streamer for moderators to access the overlay.
- **`OverlayConfig`**: Per-user configuration (colors, animation speeds, sound, goal title, and `layoutConfig` JSON string containing complete serialized `WidgetInstance[]` canvas state).
- **`AlertHistory`**: Audit log of all alerts triggered.

## State Synchronization & Alert Queue Management

The Studio Dashboard and OBS Overlay Client sync state seamlessly:
1. **Cloud Persistence**: On mount, both Studio Dashboard and OBS Overlay fetch the saved widget state from `GET /api/overlay/info/:token` or `GET /api/overlay/config/:token`.
2. **Auto-Save & Real-Time Sync**: When any widget layout, toggle, text, color, or sound setting is changed in the Studio, a debounced `POST /api/overlay/config` request is sent to the server, and a `save-studio-state` event is emitted via Socket.io to keep connected OBS browser sources perfectly in sync without page reloads.


The OBS Overlay Client maintains an internal **Alert Queue Service**:
1. Incoming `trigger-alert` events are pushed to an array queue.
2. If no alert is currently playing, the queue pops the next alert and sets active display state.
3. Audio plays on entrance. Confetti animation fires for sub events.
4. After `durationMs` elapses, the widget plays the exit animation, clears active state, and pops the next queued item.

### Alert Tier Matching
The Studio supports multiple Sub Alert widgets, each with a `triggerEventType`, `triggerMinAmount`, `triggerMinMonths`, and `triggerTier`. When an event fires, `matchAlertToWidget` scores all visible alert widgets and activates the highest-scoring match, enabling tier-specific GIFs, sounds, and text templates per alert type.
